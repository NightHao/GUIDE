## 2. Flow of the Diagram (Step by Step)

1. **Entering matching process**  
   The EV transitions into the matching phase (often triggered by a control pilot transition to an active charging state).

2. **Discovery of the connected communication nodes**  
   - The EV enumerates all communication nodes visible on the physical link.  
   - Multiple EVSEs may be detected (e.g., in scenarios with cross-talk).

3. **Signal strength measurement**  
   - The EV measures how strongly each communication nodeâs signal is received.

4. **Signal strength analysis**  
   - The EV compares measured signals. Three possible outcomes:  
     a) **Low signal attenuation** â strong signal from one node.  
     b) **Ambiguous result** â multiple nodes with uncertain or similarly strong signals.  
     c) **High signal attenuation** â no suitable (or extremely weak) signals.

5. **Branching into three results**  
   - If **low signal attenuation** is observed:  
     â The EV sets **Result = âEVSE_FOUND.â**  
   - If **ambiguous** measurement is observed:  
     â The EV sets **Result = âEVSE_POTENTIALLY_FOUND.â**  
   - If **high signal attenuation** is observed:  
     â The EV sets **Result = âEVSE_NOT_FOUND.â**

6. **Result is âEVSE_FOUNDâ**  
   - Optionally, the EV may do a sanity/validation check (depending on the scenario).  
   - Otherwise, it can declare final result as âEVSE_FOUNDâ and proceed.

   **Declared result is âEVSE_FOUNDâ**  
   â Next step: **Setup logical network**.

7. **Result is âEVSE_POTENTIALLY_FOUNDâ**  
   - The EV might treat it as âEVSE_FOUNDâ or âEVSE_NOT_FOUND,â or it can continue with an actual **Validation of matching decision**.  
   - If it decides not to validate further (per system rules), it may directly set âDeclared result is âEVSE_POTENTIALLY_FOUNDââ and treat it accordingly.  
   - Otherwise, proceed to the next diamond: **Validation of matching decision**.

8. **Result is âEVSE_NOT_FOUNDâ**  
   - If the EV cannot identify a valid EVSE, or the signal is too weak to be recognized, the EV declares âEVSE_NOT_FOUND.â  
   - **Declared result is âEVSE_NOT_FOUND.â**  
   - This immediately leads to the EV calling **D-LINK_READY.IND (No link)** and exiting the process (with failure).

9. **Validation of matching decision** (found on the âEVSE_POTENTIALLY_FOUNDâ path or optionally on the âEVSE_FOUNDâ path)  
   - The EV requests (and performs) additional confirmation with the EVSE (for instance, using the control pilot âBCB-Toggleâ procedure mentioned in Clause 9.4).  
   - **Validation result?**  
     1. **found**: The EV confirms the EVSE is correct. Proceed to âSetup logical network.â  
     2. **not found**: The EV rejects this EVSE, checks if there is a **Remaining unvalidated EVSE?**  
        - If **yes**, go to **next EVSE** and repeat.  
        - If **no**, proceed to âCall D-LINK_READY.IND (No link)â (failure).

10. **Setup logical network**  
    - Once certain of the correct EVSE, the EV sets up the low-level communication (logical network).  
    - **Data link established?**  
      - If **yes** â **Call D-LINK_READY.IND (Link established)** â exit with success.  
      - If **no** â **Call D-LINK_READY.IND (No link)** â exit with failure.

11. **Exit matching process**  
    - Either as a success (the EV is matched and ready for higher-layer communication) or as a failure (no valid link or incorrect EVSE discovered).

12. **NOTE**  
    - âError handling for Matching Process is not shown in this figure. See A.9.8.â The figure omits certain fallback or error paths (e.g., hardware faults) that are described in Annex A.9.8 in the standard.

## 3. Relation and Connection Among Entities

1. **EV** â The primary actor performing discovery, measurement, and analysis.  
2. **Communication Nodes** (EVSEs) â Possible candidates that the EV might be physically attached to.  
3. **Signal Strength Measurement** â Core step for determining whether the EV has a strong enough link to confirm âEVSE_FOUND,â or if there is ambiguity (âEVSE_POTENTIALLY_FOUNDâ), or if no match can be made (âEVSE_NOT_FOUNDâ).  
4. **Validation Process** â A separate (optional or required) mechanism that uses control pilot line toggles to double-check the physical wiring.  
5. **Logical Network Setup** â The bridging or data-layer step established once the EV confidently decides on an EVSE.  
6. **D-LINK_READY.ind** â The primitive called to indicate either a successful link or a failed link to the higher layers.

## 4. Additional Notes from ISO 15118-3

- **Under ambiguous conditions** (âEVSE_POTENTIALLY_FOUNDâ), an EV may ask the EVSE to confirm the connection via a separate pilot-line validation.  
- **Error handling** is largely omitted in Figure 10; full details appear in Annex A.9.8.  
- **Any transition**âwhether success or failureâresults in calling D-LINK_READY.IND to notify higher layers.  
- **According to Annex 9.5**, once the logical network is successfully set up, the status changes to âMatchedâ and upper-layer protocols (e.g., TLS, V2G messages) can begin binding.

### Conclusion

Figure 10 (EV point of view) outlines how the EV:  
1) **Discovers** possible EVSE nodes,  
2) **Measures** and **analyzes** each nodeâs signal strength,  
3) **Determines** whether a single, multiple, or zero EVSEs are valid matches,  
4) **Optionally validates** ambiguous results,  
5) **Sets up** a logical network with the matched EVSE, and  
6) **Exits** by either calling âD-LINK_READY.IND (Link established)â for success or âD-LINK_READY.IND (No link)â for failure.

This step-by-step process ensures the EV consistently identifies the correct EVSE in multi-charger environments and starts a reliable higher-level communication session for the charging process.
